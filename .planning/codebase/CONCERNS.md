## Tech Debt

**Generated Brain Model — Placeholder Geometry, Not Anatomically Accurate:**
- Issue: The `brain.glb` in `frontend/public/models/` was procedurally generated by `scripts/generateBrain.mjs` using scaled spheres (ellipsoids) arranged by hand-estimated coordinates. The mesh names are anatomically correct, but the geometry is not. Raycasting, hover, and visual highlight all depend on mesh geometry actually matching human brain anatomy. When a real sourced or Blender-separated model replaces this file, every mesh name in the GLB must be re-validated against `shared/src/meshNames.ts` (which does not exist yet).
- Files: `scripts/generateBrain.mjs`, `frontend/public/models/brain.glb`
- Impact: Interaction code written against ellipsoid geometry will need re-testing against a real anatomical model. Raycasting accuracy is unreliable on overlapping ellipsoids.
- Fix approach: Follow Plan 01-01 sourcing steps. After obtaining the real GLB, run `npx tsx scripts/validateGlb.ts` and update `shared/src/meshNames.ts` to match the actual mesh names.

**`shared/src/meshNames.ts` Does Not Exist Yet:**
- Issue: `shared/src/index.ts` contains stub comments: `// Plan 01-01 will add: export * from './meshNames.js'` and `// Plan 01-03 will add: export * from './regionMetadata.js'`. These exports are not present. The canonical mesh name list and region metadata are unwritten. The store (`useBrainStore.ts`) references `BrainRegion` from `@braintutor/shared`, but no region data populates it at runtime.
- Files: `shared/src/index.ts`, `shared/src/types.ts`
- Impact: All interaction code in Phase 2 that looks up regions by mesh name will fail until `meshNames.ts` and `regionMetadata.ts` are authored. The `BrainState` store fields `selectedRegion` and `hoveredRegion` always remain `null`.
- Fix approach: Complete Plan 01-01 (GLB + `meshNames.ts`) and Plan 01-03 (region metadata JSON).

**`scripts/validateGlb.ts` Missing — Only `generateBrain.mjs` Exists:**

The only script present is `scripts/generateBrain.mjs`. There is no automated guard against a replacement GLB introducing incorrectly named or unnamed meshes.
- Files: `scripts/` directory
- Impact: Without the validator, replacing `brain.glb` silently breaks name-to-region mapping. The root `package.json` already references `"validate:glb": "npx tsx scripts/validateGlb.ts"` — running it will fail.
- Fix approach: Implement `scripts/validateGlb.ts` as specified in Plan 01-01, Task 2.

## Known Bugs

**`onCreated` WebGL Error Detection Is Incomplete:**
- Symptoms: In `BrainScene.tsx`, `onCreated={({ gl }) => { if (!gl) setModelError(true) }}` attempts to detect WebGL context failure. However, `gl` from R3F's `onCreated` callback is the WebGL renderer instance — it will not be falsy on context loss; it will simply stop rendering. This check never triggers.
- Files: `frontend/src/components/brain/BrainScene.tsx` (line 67)
- Trigger: WebGL context loss (device GPU pressure, too many contexts, driver issue).
- Workaround: None currently. The app silently shows a black canvas with no user feedback. Add a `gl.domElement.addEventListener('webglcontextlost', ...)` listener inside `onCreated` to detect actual context loss.

**`NoModelOverlay` Condition Logic Has a Gap:**
- Symptoms: The overlay shows when `!modelLoaded && modelError === false`. If `modelError` becomes `true` but `modelLoaded` is still `false` (i.e., GLB load fails), the overlay disappears but no error state is shown to the user. The canvas renders as a blank dark background with no explanation.
- Files: `frontend/src/components/brain/BrainScene.tsx` (line 92)
- Trigger: GLB file missing or network error fetching `/models/brain.glb`.
- Workaround: None. Users see an empty dark canvas with no guidance.

**`BrainModel.onError` Is Never Called:**
- Symptoms: `BrainModel` accepts an `onError` prop but the component uses `useGLTF`, which throws a Promise (Suspense) on load and throws an Error on failure. The `onError` prop is passed but there is no error boundary wrapping `BrainModel` inside `BrainScene`. If the GLB fetch fails, React will throw an unhandled error to the nearest error boundary — `onError` is never invoked, so `setModelError(true)` is never called.
- Files: `frontend/src/components/brain/BrainModel.tsx`, `frontend/src/components/brain/BrainScene.tsx`
- Trigger: Serving the app with no `brain.glb` present at `/models/brain.glb`.
- Workaround: Add an `<ErrorBoundary>` component around `<BrainModel>` in `BrainScene.tsx` that calls `onError` on catch.

**Material Cloning Runs on Every `scene` Change:**
- Symptoms: In `BrainModel.tsx`, the `useEffect` clones all materials on `[scene, onMeshMap]` dependency change. `onMeshMap` is created with `useCallback` but without dependencies in `BrainScene.tsx` (line 54), so it is stable. However, if `scene` is ever re-assigned by `useGLTF`'s internal cache invalidation, all materials are re-cloned and the previous clones are not disposed — creating a memory leak.
- Files: `frontend/src/components/brain/BrainModel.tsx` (lines 21-39)
- Trigger: Only on GLB cache invalidation (rare in production, possible in hot-reload dev).
- Workaround: Add `THREE.Material.dispose()` calls in the `useEffect` cleanup function for previously cloned materials.

## Security Considerations

**CORS Allows All Localhost Origins Without Credential Restriction:**
- Risk: `backend/src/index.ts` configures CORS with `origin: ['http://localhost:5173', 'http://127.0.0.1:5173']`. This is development-appropriate, but when deployed to production the same file will need updating. There is no environment-based CORS configuration — production deployment risks either overly restrictive CORS (breaking the frontend) or forgotten wide-open CORS.
- Files: `backend/src/index.ts` (lines 6-9)
- Current mitigation: Localhost-only list is safe for local development.
- Recommendations: Move CORS origins to an environment variable (`ALLOWED_ORIGINS`). Read `process.env.ALLOWED_ORIGINS?.split(',')` at startup so production and dev use different lists without code changes.

**No API Key Protection on Backend Routes:**
- Risk: Future routes (`POST /api/explain`, `POST /api/tts`) will call Claude API and ElevenLabs API using keys stored in environment variables. The backend has no authentication, rate limiting, or API key validation middleware configured. Once deployed, any client could call these routes and exhaust API quotas.
- Files: `backend/src/index.ts`
- Current mitigation: Routes do not exist yet.
- Recommendations: Before Phase 4 routes go live, add Fastify rate limiting (`@fastify/rate-limit`) and consider a shared secret header check for the frontend-to-backend call.

**`window.__brainMeshes__` Debug Leak in Production:**
- Risk: `BrainScene.tsx` (line 57) exposes internal mesh names on `window.__brainMeshes__` for console debugging. This is intentional for manual testing but should be removed or gated behind a dev-mode check before any public deployment.
- Files: `frontend/src/components/brain/BrainScene.tsx` (line 57)
- Current mitigation: Only exposes mesh names (non-sensitive).
- Recommendations: Wrap in `if (import.meta.env.DEV)` guard or remove before production build.

## Performance Bottlenecks

**Generated GLB Uses Uncompressed Geometry (No Draco):**
- Problem: `generateBrain.mjs` writes GLB via `@gltf-transform/core`'s `NodeIO.writeBinary()` with no Draco compression. The generated `brain.glb` is ~27KB (small due to procedural geometry), but a real sourced model with 40-60 separate anatomy meshes may be 10-80MB uncompressed.
- Files: `scripts/generateBrain.mjs`, `frontend/public/models/brain.glb`
- Cause: No `draco()` transform applied in the `NodeIO` pipeline. Plan 01-01 specifies using `gltf-transform optimize --compress draco` as a post-processing step, but the generator itself does not enforce this.
- Improvement path: When replacing the generated model with a real GLB, apply Draco compression via `gltf-transform optimize brain-raw.glb brain.glb --compress draco`. Target final file size < 5MB.

**No Mobile Performance Testing:**
- Problem: The R3F Canvas runs at 60fps with ambient + directional lighting and full OrbitControls damping. No device capability detection or LOD (level-of-detail) strategy exists.
- Files: `frontend/src/components/brain/BrainScene.tsx`
- Cause: Early-phase scaffold with no mobile optimization concern yet. Dual WebGL context (Three.js + Live2D in Phase 5) is a documented risk that has not been tested on mid-range mobile.
- Improvement path: Add `dpr` (device pixel ratio) cap to the `<Canvas>` component. Evaluate a low-poly LOD model for mobile. Begin dual-WebGL context testing on Android/iOS when Phase 5 work starts.

## Fragile Areas

**Mesh Name → Region Metadata Coupling Is Entirely Manual:**
- Files: `shared/src/types.ts`, `shared/src/index.ts` (stub comments), `scripts/generateBrain.mjs`
- Why fragile: The contract between GLB mesh node names and `BrainRegion.id` is enforced only by `scripts/validateGlb.ts` (not yet written). If a mesh name in the GLB differs by even one character from the key in `regionMetadata.ts`, region lookup silently returns `undefined` — no TypeScript error, no runtime error, just missing interaction.
- Safe modification: Always run `npx tsx scripts/validateGlb.ts` after any GLB replacement. When authoring `regionMetadata.ts`, import from `MESH_NAMES` to enforce the type constraint (`BrainRegion.id` must be `MeshName`).
- Test coverage: No test covers the GLB-to-metadata name contract. `store.test.ts` uses a hand-coded mock region; `brainScene.smoke.test.tsx` mocks `useGLTF` entirely.

**`BrainModel` `useEffect` Has Unstable Dependency (`onMeshMap`):**
- Files: `frontend/src/components/brain/BrainModel.tsx` (line 39)
- Why fragile: The effect dependency array includes `onMeshMap`. If any parent refactor passes a non-memoized callback here, the effect (and material cloning) re-runs on every render — re-cloning all materials, re-building the mesh map, calling `setModelLoaded(true)` again, and leaking the old cloned materials. The current `useCallback` in `BrainScene.tsx` prevents this, but no lint rule or TypeScript constraint enforces it.
- Safe modification: Keep `handleMeshMap` wrapped in `useCallback` in any component that passes it to `BrainModel`. Consider accepting the mesh map as an imperative ref callback instead of a state setter to remove the dependency.
- Test coverage: Not tested. Smoke test mocks `useGLTF` with a non-functional traverse stub.

**Phase Validation Wave 0 Is Incomplete:**
- Files: `frontend/src/test/brainScene.smoke.test.tsx`, `.planning/phases/01-3d-foundation/01-VALIDATION.md`
- Why fragile: The validation document lists `shared/src/__tests__/regionMetadata.test.ts` as a Wave 0 requirement, but this file does not exist. The `01-VALIDATION.md` frontmatter shows `wave_0_complete: false` and `nyquist_compliant: false`. The root `package.json` test script runs `pnpm --filter shared test --run`, but `shared` has no `test` script in its `package.json` and no test files — this command will fail.
- Safe modification: Do not run root `pnpm test` until `shared/src/__tests__/regionMetadata.test.ts` is authored and `shared/package.json` adds a `test` script.
- Test coverage: Shared package has zero test coverage.

## Scaling Limits

**No Vector Database or RAG Infrastructure Yet:**
- Current capacity: None — Phase 3 (RAG Knowledge Base) has not been implemented.
- Limit: When Phase 3 begins, ChromaDB will be run locally. ChromaDB local mode does not scale to multiple concurrent users without a server deployment. LangChain.js + Transformers.js embedding runs in-process on the Node.js backend — CPU-bound embedding on the backend will block the event loop for large document indexing.
- Scaling path: Run ChromaDB as a separate Docker service. Move embedding to a worker thread or offload to a dedicated embedding service.

**Session Context Accumulates Without Bounds:**
- Current capacity: Not implemented yet (Phase 4 feature).
- Limit: Once Phase 4 implements session tracking, unbounded context accumulation will cause Claude API calls to fail when the total prompt exceeds the context window (~200K tokens for Claude 3.5 Sonnet). The STATE.md documents this as a known concern.
- Scaling path: Implement sliding window with token counting as specified in PITFALLS.md (Pitfall 7). Keep last 4-6 exchanges and summarize earlier history into a compact session state struct.

## Dependencies at Risk

**`@react-three/fiber` v9 + React 19 — Very New Combination:**
- Risk: `@react-three/fiber` v9 adds React 19 support. React 19 concurrent features (Suspense, transitions) interact with R3F's render loop in ways that may surface edge-case bugs not yet documented. Community adoption of this combination is minimal as of early 2026.
- Impact: Unexpected render behavior during Suspense boundary transitions, particularly when the GLB is loading and the `<Loader>` component is shown inside the Canvas.
- Migration plan: Monitor `@react-three/fiber` changelogs. The existing Suspense + `useGLTF` pattern is the standard approach — no immediate migration needed, but keep dependencies pinned to tested versions.

**`three` v0.183 — Rapid Release Cadence:**
- Risk: Three.js releases minor versions frequently (every 3-4 weeks). Breaking changes in geometry APIs, `GLTFLoader`, or `DRACOLoader` paths have historically appeared in minor versions. `@types/three` is a separate package that may lag behind the actual library.
- Impact: Future Three.js upgrades for bug fixes could break `useGLTF` integration or the `DRACOLoader` path configured in `scripts/validateGlb.ts`.
- Migration plan: Pin `three` and `@types/three` to matching minor versions. Upgrade deliberately with changelog review.

**`@gltf-transform/core` Used Only in Build Scripts — Not Audited for Web:**
- Risk: `@gltf-transform/core` v4 is listed as a frontend `dependency` (not `devDependency`) in `frontend/package.json`. It is only used in `scripts/generateBrain.mjs` (a Node.js script). This means the library is bundled into the frontend output unnecessarily, increasing bundle size.
- Impact: Vite will attempt to bundle `@gltf-transform/core` for the browser even though it is never imported from any frontend source file. This may cause build warnings or inflate the chunk size.
- Migration plan: Move `@gltf-transform/core` to `devDependencies` in `frontend/package.json`, or move it to the root-level `devDependencies`.

## Missing Critical Features

**No Error Boundary in the React Tree:**
- Problem: The app has no `<ErrorBoundary>` component. If `useGLTF` throws a non-Promise error (e.g., malformed GLB, network failure), React will crash the entire component tree with an unhandled error rather than showing a fallback UI. The `onError` prop on `BrainModel` cannot catch this without an error boundary.
- Blocks: Any resilient error handling for GLB load failures, future API call failures, or rendering errors.

**No Environment Variable Validation at Startup:**
- Problem: Future phases will require API keys (`ANTHROPIC_API_KEY`, `ELEVENLABS_API_KEY`). The backend has no startup validation that checks for required environment variables. A missing key will cause a runtime error deep in a request handler, not at startup.
- Blocks: Reliable deployment and early failure detection.

**No Loading State for Backend API Calls:**
- Problem: The frontend has no `lib/api.ts` or equivalent HTTP client. When Phase 4 adds the `ExplainPanel` with SSE streaming from `/api/explain`, there is no established pattern for API calls, error handling, or loading states.
- Blocks: Phase 4 frontend implementation — the pattern needs establishing before building the explanation UI.
